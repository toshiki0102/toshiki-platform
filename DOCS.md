# Toshiki Platform — システム仕様書

> 非エンジニアの方にも読んでいただけるよう、できるだけ平易な言葉で記述しています。

---

## 1. このシステムは何をするもの？

個人で撮影した写真（風景・ポートレートなど）をインターネット上で公開するギャラリーサイトです。

| 機能 | 誰でも使える？ |
|---|---|
| 写真を見る・タグで絞り込む | はい（ログイン不要） |
| 写真をアップロードする | いいえ（管理者のみ） |
| 写真を削除する | いいえ（管理者のみ） |

---

## 2. 使っているサービスと役割

```
ユーザー
  ↓ アクセス
Vercel（サイトを動かす場所）
  ↓ データが必要なとき
Supabase（データ・写真ファイル・ログイン管理）
  ↓ ログイン処理のとき
Google Cloud（本人確認）
```

| サービス | 役割 | AWS で例えると |
|---|---|---|
| **Vercel** | Next.js アプリのホスティング・自動デプロイ | EC2 + CloudFront + CodePipeline |
| **Supabase** | DB・ファイル保存・認証 | RDS + S3 + Cognito |
| **Google Cloud** | Google アカウントでの本人確認 | （外部サービス） |

## 5. データの構造（データベース）

Supabase の PostgreSQL に以下の3つのテーブルがあります。

### photos（写真）
| カラム | 内容 | 例 |
|---|---|---|
| id | 写真の一意な識別子 | `abc123...` |
| title | タイトル | `夕暮れの海` |
| description | 説明文 | `null`（現在未使用） |
| storage_path | 写真ファイルの保存場所 | `1234567_xyz.jpg` |
| category | カテゴリ | `風景` |
| created_at | 登録日時 | `2025-01-01 12:00:00` |

### tags（タグ）
| カラム | 内容 | 例 |
|---|---|---|
| id | タグの一意な識別子 | `tag123...` |
| name | タグ名 | `夕焼け` |

### photo_tags（写真とタグの紐付け）
| カラム | 内容 |
|---|---|
| photo_id | どの写真か |
| tag_id | どのタグか |

> 1枚の写真に複数のタグをつけられるよう、中間テーブルで管理しています。

## 6. 主要な機能のデータフロー

### ギャラリー表示（トップページ）

```
1. ユーザーが https://... にアクセス
2. Vercel がページを生成
3. Supabase の photos テーブルから写真一覧を取得
   （タグ絞り込みがある場合は photo_tags で絞り込み）
4. 写真を新着順にグリッド表示
5. 画像ファイルは Supabase Storage から直接配信
```

> キャッシュ: 1時間ごとに最新データを取得（ISR）。アップロード・削除時は即時更新。

---

### Google ログイン

```
1. ユーザーが「Google でログイン」ボタンを押す
2. Supabase が Google のログイン画面に転送
3. Google が本人確認 → Supabase に「OK」と通知
   （Google に登録した Supabase のコールバック URL: .supabase.co/auth/v1/callback）
4. Supabase がセッション（ログイン状態）を作成
5. /admin にリダイレクト
```

---

### 写真アップロード

```
1. 管理者が /admin/upload でファイル・タイトルを入力して送信
2. サーバー側で認証チェック（ログインしているか確認）
3. 画像ファイルを Supabase Storage にアップロード
4. photos テーブルにタイトル・カテゴリ・ファイルパスを保存
5. photo_tags テーブルに選択したタグを保存
6. ギャラリーのキャッシュを即時クリア（トップページに反映）
7. /admin にリダイレクト

※ 手順4が失敗した場合は、手順3でアップロードしたファイルを自動削除（孤立ファイル防止）
```

---

### 写真削除

```
1. 管理者が削除ボタンを押す（確認ダイアログあり）
2. サーバー側で認証チェック
3. Supabase Storage からファイルを削除
4. photos テーブルからレコードを削除
   （photo_tags は CASCADE で自動削除）
5. ギャラリーのキャッシュを即時クリア
```

## 7. セキュリティ（RLS）の仕組み

### RLS（Row Level Security）とは？

データベースの「行レベルのアクセス制御」です。  
「誰が」「何を」「できるか」をテーブルの行単位で制御できます。

わかりやすく言うと：

```
鍵のかかった倉庫をイメージしてください。
  一般の人 → 中を見ることはできる（読み取りOK）
  管理者   → 荷物を入れたり出したりできる（書き込みOK）
```

### このプロジェクトの RLS 設定

| テーブル | 一般ユーザー（未ログイン） | 管理者（ログイン済み） |
|---|---|---|
| photos | 読み取りのみ | 読み取り・書き込み・削除 |
| tags | 読み取りのみ | 読み取り・書き込み |
| photo_tags | 読み取りのみ | 読み取り・書き込み・削除 |

### なぜ重要？

**RLS がない場合のリスク：**
```
悪意のある人が API を直接叩いて
→ 全写真を削除
→ 不正なデータを書き込む
→ 他のユーザーの情報を取得
```

**RLS がある場合：**
```
データベースのレベルで弾くため
→ アプリのコードを迂回されても安全
→ 認証トークンがなければ書き込み不可
```

### 認証トークンの検証

管理者かどうかの確認は `getClaims()` で行っています。

```
getClaims() → JWT トークンの署名を毎回検証
→ 改ざんされたトークンは弾かれる
```

> `getSession()` はサーバー側で使うと JWT を検証しないため、このプロジェクトでは使用していません。

|---|---|
